# Occulta — Technical Audit & Improvement Plan

**Date:** 2026-06-10 09:00
**Branch audited:** `v1.8.0/secure-mode` (HEAD `9d3fe3b`)
**Scope:** Full-repo audit — architecture, code quality, security, testing, performance, dependencies, DevEx, documentation. Analysis only; no code modified.
**Method:** Direct source reading of the SecureMode, Key/Crypto management, ShareExtension boundary, and Shamir code in depth; structural pass over the rest. All findings cite file:line and are labeled **[F]** (verified fact) or **[J]** (judgment).

---

## Executive Summary

**Overall health: B+.** Occulta is a serverless iOS cryptographic contact book in an unusually healthy state for a solo-developed security product: the crypto core is disciplined (domain-separated HKDF everywhere, SE-bound keys, AAD on local encryption), 449 unit tests cover the protocol logic, and the design documentation (in-code crypto review checklists, a 69-entry bug tracker, threat-model docs) is better than most funded teams produce.

**Top 3 risks:**
1. No CI of any kind — 449 tests that nothing enforces, in a codebase that just went through a 30-bug key-rotation refactor.
2. Decryption failures are silently swallowed at ~88 call sites (`String.decrypt()` returns `""`), so the exact failure mode key rotation can produce — wrong-key ciphertext — is invisible.
3. The PIN brute-force lockout is wall-clock based and can be bypassed by changing the device clock, squarely inside the app's stated coercion threat model.

**Top 3 opportunities:**
1. Cache the hybrid local-DB key — today every decrypted field pays a Secure Enclave ECDH round-trip, including inside sort comparators.
2. Replace the global `Data.decrypt()`/`String.decrypt()` extensions with an injected crypto provider — simultaneously fixes the silent-error problem and unlocks the activation round-trip tests the test suite itself says it can't write.
3. A one-hour CI setup converts the existing test suite into an actual safety net.

## Phase 1 — Repo Map

**Purpose.** Offline-first, serverless iOS contact book where UWB proximity (≤ 25 cm, NearbyInteraction) gates in-person public-key exchange. Users then encrypt files for contacts (AES-256-GCM, hybrid P-256 + ML-KEM-1024) and share via any channel. Shipping on the App Store (README.md:5); maturity is "production app, actively developed" — current branch `v1.8.0/secure-mode` is mid-feature (duress-PIN/decoy-layer system).

**Stack.** Swift 5 / SwiftUI / SwiftData, iOS 18.6 deployment target, Xcode project with one SPM dependency (apple/swift-crypto 4.2.0, pinned in `Package.resolved`). Apple crypto only otherwise: CryptoKit, Security.framework, Secure Enclave, NearbyInteraction, MultipeerConnectivity. ~34.8k lines of Swift across 4 targets: app, ShareExtension, OccultaPreview (Quick Look), tests.

**Key directories.**

| Path | Role |
|---|---|
| `Occulta/Services/` | Core managers: `Key+Manager.swift` (1,423 ln — all SE key lifecycle), `Crypto+Manager.swift`, `Contact+Manager.swift` (1,286 ln), `Exchange+Manager.swift` (UWB/MPC exchange) |
| `Occulta/Features/SecureMode/` | Duress-PIN state machine (`Manager+Security.swift`, 1,267 ln), PIN verifier crypto, deniable layer-store blob |
| `Occulta/Features/Vault/` | Encrypted vault + in-house Shamir secret sharing + shard custody protocol |
| `Occulta/Features/Forward+Secrecy/`, `PostQuantum/`, `IdentityChallenge/` | Prekey FS, ML-KEM hybrid, signed identity challenges |
| `Occulta/Features/ShareExtension/` + `ShareExtension/` | App-group share index (deliberately isolated crypto boundary) |
| `Occulta/UI/Tabs/` | SwiftUI views; Contacts has parallel `v1/` and `V2/` trees |

| `Docs/`, `Audit/`, `Friction/` | Design docs, bug tracker (69 bugs), release security checklist |

**Control flow.** `Occulta/App/OccultaApp.swift` builds the SwiftData container, stamps file protection, applies `PRAGMA secure_delete`, wires six `@Observable` managers into the environment. `RootView` gates everything behind an `AppScreen` phase machine (covered → pinRequired → unlocked); inbound `.occ` files arrive via `onOpenURL` or share-extension handoff and are queued as raw bytes until PIN depth is known.

**Surprises.** Good: in-code `CRYPTO_REVIEW_CHECKLIST` blocks (e.g. `ShamirSecretSharing.swift:9-48`); the forensic-deniability reasoning is carried into code comments at the exact decision points. Bad: no CI, no lint config; CLAUDE.md's claim of "no external package manager (no CocoaPods, SPM, or Carthage)" is factually wrong (swift-crypto is an SPM dependency); the documented test CLI command in CLAUDE.md is literally missing ("or via CLI:" followed by nothing).

## Phase 2 — Audit Report

Severity ordering within each dimension. **[F]** = verified fact, **[J]** = judgment.

### Security

**SEC-1 · High [F] — PIN lockout is wall-clock based and bypassable by changing the device clock.**
Where: `Occulta/Features/SecureMode/Manager+Security.swift:833` (`Date.now < expiry`) and `:864` (`Date.now.addingTimeInterval(delay)`).
Why it matters: the PIN is 6 digits (`Occulta/UI/Tabs/SecureMode/PINEntry.swift:70`), and because verifiers are SE-bound (good — no offline brute force), the escalating lockout (max 24 h, `Manager+Security.swift:872-891`) is the *only* throttle. An adversary holding the unlocked device — precisely the coercion scenario this feature is built for — can open Settings, set the clock forward, and reset every lockout instantly. Automated input (HID/switch-control scripting) makes a 10⁶ keyspace plausible over days. The threat model docs don't list this as accepted.

**SEC-2 · Medium [F] — Lockout counters silently reset to zero on decode failure, and are encrypted under the rotating local-DB key.**
Where: `Occulta/Features/SecureMode/AppLayerConfig+Model.swift:391-397` (`return 0` on any failure); the values are written via `.encrypt()` (local DB hybrid key). Secure Mode activation/deactivation rotates that key and re-encrypts contacts and vault fields (`Manager+Security.swift:466-498`) but no re-encryption of `lockoutCountEncrypted`/`lockoutExpiryEncrypted` was found in the rotation sequence — after rotation they become undecryptable and read as 0/nil. Practical impact is small (rotation requires a correct PIN), but an in-progress lockout vanishes after any state transition, and any single-field DB corruption resets the brute-force counter. Combined with SEC-1 it weakens the throttle further.

**SEC-3 · Medium [F] — `Crypto.sign(data:)` returns human-readable error strings through the signature channel.**
Where: `Occulta/Services/Crypto+Manager.swift:126-151` — returns `"Key or data is missing"`, `"Algorithm is not supported"`, etc., as the `String` result; rendered uppercased as the signature in `Occulta/UI/Tabs/Sign/Sign.swift:75`. A user could publish an "error message" believing it's a signature. Mitigated today by the `signature` feature flag being `false` (`Occulta/features.plist`), but the code is live.

**SEC-4 · Low [F] — Release-build `print()` of exchange events and peer identity material.**
Where: `Occulta/UI/Tabs/Contacts/Key+Exchange/KeyExchange.swift:58` and `:74` print received peer public keys *outside* any `#if DEBUG` guard (the adjacent `debugPrint`s are guarded); also `Occulta/Services/Exchange+Manager.swift:613`, `Occulta/Features/Features.swift:16`. Public keys aren't secrets, but the project's own bar is "forensic trace clean" (CLAUDE.md): timestamped evidence that a key exchange occurred is exactly the kind of trace the rest of the codebase works hard to avoid.

**SEC-5 · Low [F] — Unbounded inbound file read into memory.**
Where: `Occulta/App/OccultaApp.swift:431` — `URLSession.shared.data(from: fileLocation)` loads any `.occ` fully into RAM before validation. A multi-GB file shared into the app is a crash, not an exploit, but there's no size guard anywhere on the inbound path.

**SEC-6 · Low [F/J] — Shamir implementation: two hardening gaps, otherwise correct.**
Where: `Occulta/Features/Vault/ShamirSecretSharing.swift`. (a) `reconstruct()` accepts ≥ 2 shares without knowing `k` (`:136-137`) — below-threshold input silently yields a wrong secret; the threshold is enforced one layer up, which the comments honestly document, but the share wire format could carry `k` for defense in depth. (b) `gfMul` branches on secret-dependent bits (`:163-175`) — not constant-time; low risk since it runs locally on-device, not in a remote-observable path. The math itself (GF(2⁸)/0x11B, x=0 share rejection at `:143`, duplicate-x rejection, non-zero leading coefficient) checks out.

**What security gets right.** No hardcoded secrets; no network surface at all; five SE keys with distinct tags, distinct HKDF info strings, and access policies matched to purpose (documented inventory at `Key+Manager.swift:52-72`); 65-byte validation before every `SecKeyCreateWithData` (`Key+Manager.swift:358`); MPC session uses `encryptionPreference: .required` (`Exchange+Manager.swift:121`); the share extension's crypto isolation boundary (`ShareExtension/ShareViewController.swift:8-10`, `ShareIndexKeyManager.swift`) is a genuinely good design; the staged-key rotation with WAL-checkpoint ordering (`Manager+Security.swift:500-515`) shows real understanding of the failure modes; locked-state inbound files are queued as raw bytes until PIN depth is known (`OccultaApp.swift:439-445`).

### Code Quality

**CQ-1 · High [F] — Decryption failures are silently swallowed at scale.**
Where: `Occulta/Services/Crypto+Manager.swift:168-199` — `String.decrypt()` returns `""` on *any* error; `Data.decrypt()` returns `nil` via `try?`. 88 call sites across models, managers, and UI. Why it matters: wrong-key ciphertext — the exact artifact of an interrupted key rotation, the subject of Bugs 37–69 in `Docs/Features/Secure Mode/bugs.md` — renders as an empty contact name, not an error. The user sees blank contacts; the developer gets no signal; data corruption and "field was empty" are indistinguishable. Several security reads inherit the pattern: `readCoercerBaseDepth`, `readLockoutCount` etc. fall back to defaults on failure (some intentionally conservative, per comments, but the conflation is systemic).

**CQ-2 · Medium [J] — `OccultaApp.swift` (809 ln) is accreting unrelated responsibilities.** App bootstrap, `RootView`, URL routing, inbound-file pipeline, share-session processing, identity-challenge sheet wiring, and shard handling all live in one file (e.g. the inbound pipeline at `:367-548`). It's the file every feature must touch; merge friction and regression risk concentrate here. Same tendency, lesser degree: `Contact+Manager.swift` (1,286 ln) mixes CRUD, key rotation, bundle encrypt/decrypt, and trustee selection — though extensions partially segment it.

**CQ-3 · Medium [F] — Near-identical ECDH(priv, G) → HKDF derivation repeated five times in Key+Manager.**
Where: `createLocalEncryptionKey` `:131`, `deriveVaultKey` `:715`, `deriveShardCustodyKey` `:815`, `deriveRecoveryBufferKey` `:854`, `deriveSecureModeKey` `:888` — each ~25 lines differing only in key tag, salt source, and info string. A single parameterized helper would shrink ~125 lines to ~30 and make the per-key differences (the actual security surface) explicit. (The duplicated generator-point constant in `ShareIndexKeyManager.swift:33` is *intentional* extension isolation and documented — leave it.)

**CQ-4 · Low [F] — Parallel v1/V2 contact UI trees.** `ContactsV2` is the live tab (`OccultaApp.swift:326`); v1's `Contacts` list appears only in its own preview (`Contacts.swift:198`), yet `ManageContacts` (`Settings.swift:29`) and `ComposableMessage` (`OccultaApp.swift:197`) keep parts of `v1/` alive. Which v1 files are dead is not discoverable without tracing — a deletion-candidate inventory is needed before anything is removed.

**CQ-5 · Low [F] — Minor hygiene.** Known-typo public API `retrivePublicKey` kept deliberately with a doc note (`Key+Manager.swift:189-191`); `try?` on forensically-load-bearing seed writes in `Security.init` (`Manager+Security.swift:154-204`) — if those silently fail, the "config row always exists" invariant the deniability design depends on quietly doesn't hold; stray-copy filename `SHARE_EXTENSION_PLAN 16.32.42.md` in `Occulta/Features/ShareExtension/`.

### Performance

**PERF-1 · High [F] — Every field decryption re-derives the DB key through the Secure Enclave.**
Where: `Crypto+Manager.swift:67-79` calls `createHybridLocalEncryptionKey()` per decrypt; that function (`Key+Manager.swift:431-457`) performs SE key retrieval + SE ECDH + Keychain read + HKDF *every time* — no caching anywhere. The blast radius: `Occulta/UI/Tabs/Contacts/v1/Contacts.swift:209-223` calls `.decrypt()` **inside sort comparators** (O(N log N) comparisons × 2–4 decrypts each); `ContactDetailV2.swift:30-34` decrypts in computed properties re-evaluated per SwiftUI render; `syncShareIndex` decrypts every contact name on every save and scene activation (`ContactManager+ShareIndex.swift:53-54`). SE round-trips are on the order of milliseconds; at a few hundred contacts this is thousands of SE calls per list render — UI jank that worsens linearly with the user's contact count, on the app's primary screen. Caveat: the no-cache design may be a deliberate "key never rests in memory" stance, but the derived key already lives transiently per call, and a scene-phase-scoped cache (cleared on background, invalidated on rotation) would preserve nearly all of the security property. Decide explicitly; today it looks unintentional.

**PERF-2 · Low [F] — Full share-index rebuild on every contact save** (`ContactManager+ShareIndex.swift:14-18`) — self-documented as a deliberate simplicity trade-off; fine at this scale. Healthy otherwise; no other hot loops found.

### Testing

**TEST-1 · High [F] — No CI.** No `.github/`, no fastlane, no scripts, no test plan automation. 449 tests exist and are well-organized, but nothing runs them on push. For a crypto product whose recent history is "69 tracked bugs, several of them data-loss-grade, found during a key-rotation refactor," an unenforced suite is a safety net lying on the ground.

**TEST-2 · Medium [F] — The most dangerous code path can't assert its core invariant.** `OccultaTests/SecureMode/SecureModeActivationTests.swift:8-21` states it directly: because `ContactManager` reaches crypto via `Manager.Crypto()` → `Manager.Key()` (hard-wired through the `Data.decrypt()` extensions), while `Manager.Security` uses the injected `TestKeyManager`, *post-activation field-content round-trips are untestable*. The tests fall back to blob I/O and WAL-persistence smoke checks. The single highest-risk operation in the app — re-encrypting the user's entire database — has no end-to-end "data survives" assertion. Root cause is CQ-1's architecture, not test effort.

**TEST-3 · Low [F] — UI tests are Xcode template stubs** (74 lines total in `OccultaUITests/`), and no tests were found exercising `ShareViewController` logic. Test *quality* elsewhere is genuinely good: behavioral assertions, honest architecture notes, dedicated regression tests named for bug numbers.

### Dependencies

Healthy. Single dependency (apple/swift-crypto 4.2.0 + transitive swift-asn1), pinned in `Package.resolved`, from Apple — minimal supply-chain surface. **[F]** CLAUDE.md's "no external package manager" and README's "Apple frameworks only (CryptoKit + Security.framework)" are both contradicted by the SPM reference at `Occulta.xcodeproj/project.pbxproj:1055-1057` and `import Crypto` at `Crypto+Manager.swift:14` — worth either documenting why swift-crypto is needed (only one file imports it) or removing it.

## Phase 3 — Improvement Strategy

**Theme 1: The safety net exists but isn't plugged in.**
Findings explained: TEST-1, DEV-2, DEV-3, the unchecked SECURITY_CHECKLIST. Target state: every push to a `v1.*` or `develop` branch builds and runs the simulator-safe test suite; docs state one true requirement set; releases produce a dated, filled checklist. Principle: *a control that doesn't run automatically doesn't exist.* This is the cheapest theme and multiplies the value of work already done (the 449 tests).

**Theme 2: Errors are silenced where they matter most.**
Findings explained: CQ-1, SEC-2, SEC-3, parts of CQ-5. Target state: decryption failure is *representable* — distinct from empty — at every call site that renders or persists; security-state seed writes assert or surface. Principle: *in a crypto app, "couldn't decrypt" is a first-class event, never a default value.*

**Theme 3: Crypto access is a global, not a dependency.**
Findings explained: TEST-2, CQ-1 (the extensions are the vector), PERF-1 (no seam to put a cache behind), CQ-3. Target state: one injected crypto provider per container; the `String`/`Data` convenience extensions delegated to it or removed; key derivation written once. Principle: *the seam that enables testing is the same seam that enables caching* — one refactor pays twice.

**Theme 4: The coercion threat model has one un-modeled adversary capability: the clock.**
Findings explained: SEC-1, SEC-2. Target state: lockout enforcement anchored to a monotonic source, counter survival across key rotation. Principle: anything the adversary can change in Settings is not a security boundary.

**Explicitly NOT recommending:**
- Replacing the in-house Shamir with a library — it's correct, tested, dependency-freedom is a stated project value, and the gaps (SEC-6) are hardening, not flaws.
- Constant-time GF(2⁸) arithmetic — no remote observer exists; effort exceeds the threat.
- Breaking up `Manager+Security.swift` — it's long but it's a *cohesive* state machine whose steps are sequentially documented; splitting it would scatter the 11-step rotation narrative that comments currently keep legible.
- Real UI test coverage — high maintenance cost on a fast-moving SwiftUI surface; the unit layer is where this app's risk lives.
- Deleting v1 UI now — inventory first (it's partially live), delete in M3.

**Definition of done (measurable):** CI green required to merge to `develop`; zero non-`#if DEBUG` `print()` in app targets (greppable); `String.decrypt()`/`Data.decrypt()` global extensions gone or routed through an injected provider; an activation→deactivation round-trip test asserting field plaintext equality exists and passes; lockout test simulating clock jump passes; one consistent iOS requirement across CLAUDE.md/README/pbxproj; contact list render performs ≤ 1 SE derivation per snapshot (instrumentable with signposts).

### Implementation sketches — top 3

**Task 0.1 (CI).** `macos-15` runner, Xcode 16.x via `maxim-lobanov/setup-xcode`; `xcodebuild test -project Occulta.xcodeproj -scheme OccultaTests -destination 'platform=iOS Simulator,name=iPhone 16'`. Gotchas: tests that need real SE must keep their existing skip-guards (the suite already has an SE-availability helper, `SecureModeActivationTests.swift:19-21` — verify each suite uses it); signing — set `CODE_SIGNING_ALLOWED=NO` for the test build since no device deploy; cache derived data keyed on `Package.resolved`. Start with build+test only; add the lint job (0.4) as a second step.

**Task 2.2 (key caching).** Add a `private var cachedHybridKey: (key: SymmetricKey, epoch: Int)?` behind a lock inside `Manager.Key` (or, better, in the 2.1 provider). Increment an epoch counter inside `commitStagedLocalDBKey()` and `rollbackStagedLocalDBKey()` (`Key+Manager.swift:1032, 1124`) so any in-flight cache is invalidated at the only two places the canonical key can change. Clear on `UIApplication.didEnterBackgroundNotification` to preserve the "key not at rest while backgrounded" property. Gotchas: the share extension must *not* get this cache (it doesn't use this key — verify nothing in the extension target links Key+Manager; the boundary docs say it doesn't); `Manager.Crypto()` is instantiated fresh at call sites today, so the cache must live in `Manager.Key`'s storage or a shared provider, not on the `Crypto` instance — otherwise it caches nothing.

**Task 1.1 (monotonic lockout).** Store three values on lockout: wall-clock expiry (existing), `kern.boottime + systemUptime`-derived monotonic deadline, and the attempt timestamp. In `verify()`'s lockout check, remain locked if *either* deadline is unexpired, and treat `Date.now < lastAttemptTimestamp` (clock rolled back) as still-locked. Gotchas: monotonic time resets on reboot — fall back to wall-clock-only after reboot and accept that residual (document it in `forensic-trace-avoidance.md`'s accepted-gaps section, which is the established pattern); all three fields must be encrypted like the existing counters *and* carried through rotation (Task 1.2); write the clock-jump unit test against `lockoutDelay(for:)`'s seam by injecting a `now` provider rather than calling `Date.now` inline.

## Open Questions

1. **Is the no-cache key derivation (PERF-1) a deliberate security stance or an accident?** If deliberate, the scene-phase-scoped cache is still arguably worth it, but the accepted-risk doc should record the decision either way.
2. **What is the real minimum iOS target?** pbxproj says 18.6; README sells iOS 17 and "iOS 26+ for PQ." If 18.6 is just "whatever Xcode set," lowering it is a product decision that changes the ML-KEM availability matrix.
3. **v1 UI (`ManageContacts`, the v1 Form stack, `ComposableMessage`): planned migration to V2, or permanent?** Determines whether 3.1 is a deletion or a port.
4. **Shamir share-format change (3.2): are there already user-distributed shards in the wild?** If yes, a v2 wire format needs a custody-protocol migration story and may not be worth it.
5. **Lockout policy intent:** is "first 5 attempts free" + 24 h cap the desired curve for a 6-digit PIN, given the coercion model? Wipe-after-N conflicts with deniability (a wipe is a tell), so this is a product decision.
6. **The `signature` feature flag has been off** — is the Sign/Verify tab a deprecation candidate? If so, 1.4 becomes deletion instead.
