# Forward Secrecy Guide

Commit this file alongside any PR that touches the forward secrecy implementation.

## Data model

`ForwardSecrecy` is a `Codable` struct encrypted as a single AES-GCM blob on `Contact.Profile`.  
Decrypt once at the start of an operation, mutate, re-encrypt on every write.

```swift
struct ForwardSecrecy: Codable {
    var encodedPrekeys:           [Data]? // JSON-encoded Prekey blobs received FROM this contact
    var latestPrekeysGeneratedAt: Date?   // when their last batch was created â€” stale-batch guard
    var pendingOutboundBatch:     Data?   // JSON-encoded PrekeySyncBatch waiting for proof of receipt
}
```

`encodedPrekeys` stores **plain JSON** `Prekey` blobs. They are not individually encrypted.  
`ForwardSecrecy` is AES-GCM encrypted at the struct level before being written to SwiftData.  
Double-encrypting blobs inside an already-encrypted container breaks `JSONDecoder` â€” see mistake 1.

---

## Two prekey directions

`encodedPrekeys` holds **Bob's** prekeys â€” generated by Bob, delivered to Alice, consumed by Alice when encrypting to Bob.  
Our own prekeys (Alice's, for Bob to use) live exclusively in the Secure Enclave. No local blob store.

---

## SE tag format

```
"prekey.<contactID>.<uuid>"
```

Tags are fully deterministic. Given `contactID` (from fingerprint scan) and `prekeyID` (from the bundle), the full tag is reconstructed at decrypt time with no local lookup:

```swift
let temp    = Prekey(id: prekeyID, contactID: sender.identifier, publicKey: Data())
let privKey = prekeyManager.retrievePrivateKey(for: temp)
```

`publicKey: Data()` is never read by `seTag` or `retrievePrivateKey(for:)`. It is irrelevant to the SE lookup. The temp `Prekey` exists only to produce the tag string.

No sequence component in the tag. Keys are deleted individually via `consume()` and wholesale via `deleteAllKeys(for:)` when a contact is removed.

## Batch generation â€” one trigger only

**In `decrypt`, when a `.longTermFallback` bundle is received:**

```swift
if bundle.secrecy.mode == .longTermFallback && !sender.hasPendingBatch {
    let prekeys = try prekeyManager.generateBatch(contactID: sender.identifier)
    let batch   = OccultaBundle.SealedPayload.PrekeySyncBatch(generatedAt: Date(), prekeys: ...)
    try sender.store(batch: batch)
}
```

A `.longTermFallback` means the sender exhausted our prekeys. We generate a fresh batch and store it as pending so it rides the next outbound message.

`!sender.hasPendingBatch` prevents overwriting a batch already in flight, which would orphan the SE private keys from that batch.

**`encryptBundle` never generates a batch.** Generation is purely reactive to the other side's fallback signal. The first-ever conversation starts with two fallback messages in each direction â€” both sides detect the fallback, generate, and FS is established after four messages. No bootstrap special case is needed.

**`needsReplenishment` is not a trigger.** Low SE stock is the other side's problem to signal via fallback, not ours to monitor.

## The pending batch guarantee

```
generateBatch()                     â†’ store as pendingOutboundBatch
encryptBundle()                     â†’ loadPendingBatch() attached to every SealedPayload
decrypt(.forwardSecret success)     â†’ consume() fires â†’ clearPendingBatch()
```

The batch rides every outbound message until cleared. Cleared only when `consume()` succeeds inside `decrypt` â€” that is cryptographic proof the other side received and used one of our prekeys.

No server, no delivery receipts. An unforgeable consumed prekey is the only reliable receipt available.

## SecKey lifetime â€” closure pattern

`SecItemDelete` inside `consume()` invalidates the SE item backing the `SecKey`. If ARC releases the `SecKey` after `SecItemDelete`, `CFRelease` crashes on freed memory.

The `SecKey` must go out of scope **before** `consume()` is called. The closure enforces this. The temp `Prekey` is constructed twice â€” once inside the closure to retrieve the SecKey, once outside after the closure returns to call `consume()`:

```swift
// Inside closure â€” SecKey lives here, released at brace
let decrypted: Data? = try {
    let temp    = Prekey(id: prekeyID, contactID: sender.identifier, publicKey: Data())
    let privKey = prekeyManager.retrievePrivateKey(for: temp)
    ...
    return try cryptoOps.open(bundle, using: sessKey)
    // privKey released here
}()

// Outside closure â€” SecKey is gone, safe to delete SE item
if decrypted != nil {
    let temp = Prekey(id: prekeyID, contactID: sender.identifier, publicKey: Data())
    prekeyManager.consume(prekey: temp)
}
```

The duplication is intentional and must not be collapsed into a single scope.

## SE ordering rule

All SE writes must complete before any ECDH begins.

```
CORRECT:
  generateBatch()  â†’  SE writes done, SE idle
  seal()           â†’  ECDH, AES-GCM â€” zero SE side effects

WRONG:
  seal() internally calling generateBatch() then immediately running ECDH
```

The SE `securityd` daemon produces `malloc: pointer being freed was not allocated` when SE writes and ECDH are interleaved in the same call stack.

## syncInboundPrekeys â€” direct replace

```swift
func syncInboundPrekeys(_ blobs: [Data], date: Date) throws
```

Accepts only when `date > latestPrekeysGeneratedAt`.  
On acceptance: `secrecy.encodedPrekeys = blobs`. Direct replace.

**Why direct replace is always correct:**  
A new batch legitimately arrives only after the sender exhausted the previous one. When that happens, `encodedPrekeys` is already empty â€” there is nothing to preserve or prune. Append or prune-then-append logic is dead code in this design.

## No silent security degradation

When `contactPrekey` is non-nil, the FS path was explicitly requested. Any failure throws â€” it never falls back to producing a `.longTermFallback` bundle silently.

| Error | Cause |
|---|---|
| `invalidRecipientMaterial` | long-term key is not exactly 65 bytes |
| `invalidPrekeyMaterial` | `contactPrekey.publicKey` is not exactly 65 bytes |
| `ephemeralKeyGenerationFailed` | SE could not generate the throwaway key pair |
| `keyDerivationFailed` | ECDH failed despite valid key material |

## AAD

```
computeAdditionalAuthentication() = version.rawValue.utf8 || sortedKeys(JSON(SecrecyContext))

Covered by GCM tag â€” modification causes authenticationFailure:
  version, mode, ephemeralPublicKey, prekeyID

Not in AAD (readable before key derivation â€” routing only):
  fingerprintNonce, senderFingerprint

Not in AAD (inside ciphertext â€” encrypted + authenticated by the same tag):
  SealedPayload, PrekeySyncBatch, WirePrekeys, message
```

Always use `OccultaBundle.computeAdditionalAuthentication(version:secrecy:)` on both seal and open.  
The encoder must always use `.sortedKeys` â€” without it, different encoder instances produce different byte sequences for the same struct, causing spurious `authenticationFailure` on every open.

## Wire format

```
OccultaBundle {
  version                 â€” in AAD
  secrecy {
    mode                  â€” in AAD: forwardSecret | longTermFallback
    ephemeralPublicKey    â€” in AAD: 65 bytes (empty Data for fallback)
    prekeyID              â€” in AAD: UUID of consumed prekey, or nil
  }
  ciphertext              â€” AES-GCM(SealedPayload): nonce || ct || tag
  fingerprintNonce        â€” routing, not in AAD
  senderFingerprint       â€” routing, not in AAD
}

SealedPayload (encrypted + authenticated inside ciphertext) {
  message                 â€” plaintext bytes
  prekeyBatch {           â€” nil, or sender's fresh prekeys
    generatedAt           â€” Date, for stale-batch guard
    prekeys: [WirePrekey {
      id                  â€” UUID
      publicKey           â€” 65 bytes
    }]                    â€” no contactID on the wire
  }
}
```

`WirePrekey` carries no `contactID`. The sender's internal identifier for the recipient must not travel on the wire â€” it would leak the relationship graph to a passive observer. The recipient reconstructs the full `Prekey` by supplying their own local identifier as `contactID` when decoding the batch.

## generateBatch failure handling

Throws `PrekeyError.seKeyCreationFailed` on the first SE failure. Never returns a partial batch. A partial batch is indistinguishable from a full one â€” the recipient stores it, exhausts it prematurely, and triggers an unnecessary fallback with fewer usable prekeys.

## Historical mistakes

1. **Double-encrypting prekey blobs inside ForwardSecrecy.** Individual blobs passed through `cryptoOps.encrypt()` before storage. `ForwardSecrecy` is already AES-GCM encrypted at the struct level. The second layer of encryption made `JSONDecoder` see ciphertext bytes instead of JSON â€” every pop failed silently.

2. **SE tag constructed from raw UUID instead of full tag.** `retrievePrivateKey(using: prekeyID)` passed the bare UUID to `SecItemCopyMatching`. The SE tag is `"prekey.<contactID>.<id>"`. Lookup always returned nil â€” FS decryption never succeeded.

3. **Silent fallback when FS was intended.** ECDH failure with a valid `contactPrekey` called `self.fallback(...)` silently. The caller believed it sent FS; the bundle used long-term keys.

4. **SE writes interleaved with ECDH.** `generateBatch` called inside `seal`, immediately followed by ephemeral key generation and ECDH. SE daemon produced malloc corruption.

5. **SecKey held across SecItemDelete.** `consume()` called while `SecKey` still in scope. `CFRelease` on freed SE memory â†’ crash.

6. **fatalError on nil ECDH error.** `SecKeyCopyKeyExchangeResult` can return nil without populating the error parameter. Force-unwrapping the nil error â†’ undefined behaviour.

7. **Non-deterministic AAD.** `JSONEncoder()` without `.sortedKeys` produced different key orderings across instances â†’ spurious `authenticationFailure` on every decrypt.

8. **version not in AAD.** Attacker could flip version from `.v3fs` to `.v1` with no GCM failure â€” wrong security badge displayed.

9. **Speculative batch generation in encryptBundle.** Generating when SE stock fell below threshold sent a batch the other side did not need yet. Also fired when Alice exhausted Bob's prekeys â€” not a signal Bob needed more of Alice's keys. Fixed: generation is purely reactive to `.longTermFallback`.

10. **ownPrekeys array.** Maintained a local blob store of our own prekeys so the sequence could be recovered for SE tag construction. Unnecessary â€” the tag is fully deterministic from `contactID` and `prekeyID` alone. The array introduced a collision bug where sequences advancing in one direction overwrote entries in the other.

11. **hasPendingBatch not checked before generating on fallback.** A second incoming fallback could overwrite the pending batch, orphaning the SE private keys from the first generation.

12. **Bootstrap batch generation in encryptBundle.** A special case for first-ever messages generated a batch in `encryptBundle` when no inbound prekeys existed. Unnecessary â€” the natural fallback exchange bootstraps both sides without any special-casing.
